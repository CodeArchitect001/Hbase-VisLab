# ZooKeeper 与 HMaster 关系详解

> 深入理解 HBase 高可用架构的核心：ZK 如何管理 HMaster 的 Active/Standby 状态

---

## 一、核心机制总览

HBase 依赖 ZooKeeper 实现 HMaster 的高可用（HA）。核心思想很简单：

**谁先抢到 ZK 上的独占锁，谁就是 Active HMaster。**

```
┌─────────────────────────────────────────────────────────────┐
│                      ZooKeeper 集群                          │
│                                                             │
│   /hbase/master  (Ephemeral Node - 临时节点)                │
│        │                                                    │
│        ▼                                                    │
│   ┌──────────┐    会话存活 = 节点存在                        │
│   │ Active   │    会话断开 = 节点自动删除                     │
│   │ HMaster  │                                              │
│   └──────────┘                                              │
│                                                             │
│   /hbase/backup-masters  (Persistent Node)                  │
│        │                                                    │
│        ├── Standby HMaster #1 (Watcher 监听 /master)        │
│        ├── Standby HMaster #2 (Watcher 监听 /master)        │
│        └── Standby HMaster #N                               │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 三个关键概念

| 概念 | 说明 |
|------|------|
| **Ephemeral Node（临时节点）** | 与客户端会话绑定，会话结束（崩溃/断开）时自动删除 |
| **Watcher（监听器）** | Standby HMaster 注册在 `/hbase/master` 上的监听器，节点变化时收到通知 |
| **竞争写锁** | 多个 Standby 同时尝试创建 `/hbase/master` 节点，只有一个能成功 |

## 二、正常状态：Active + Standby  coexist

```
┌──────────────┐         ┌──────────────────────────────┐
│   Client     │         │         ZooKeeper             │
└──────┬───────┘         │  ┌──────────────────────┐    │
       │                 │  │ /hbase/master        │◄───┼── Active HMaster
       │ 读写请求直接到   │  │ (Ephemeral Node)     │    │   (持有锁)
       │ RegionServer    │  └──────────────────────┘    │
       │                 │                              │
       └──────────────►  │  ┌──────────────────────┐    │
                         │  │ /hbase/backup-masters│    │
┌──────────────────┐     │  │ (Persistent Node)    │    │
│  Active HMaster  │◄────┘  │ ├── standby-1        │◄───┼── Standby HMaster #1
│  (管理元数据)     │        │ ├── standby-2        │◄───┼── Standby HMaster #2
│                  │        │ └── ...              │    │
│ • DDL 操作       │        └──────────────────────┘    │
│ • Region 分配    │                                   │
│ • 负载均衡       │                                   │
└──────────────────┘                                   │
                                                       │
┌──────────────────┐                                   │
│ Standby HMaster  │── Watcher 注册在 /hbase/master 上  │
│ (待命状态)        │   监听节点是否存在                  │
│                  │                                   │
│ • 不处理请求      │                                   │
│ • 等待成为Active  │                                   │
└──────────────────┘                                   │
                                                       │
┌──────────────────┐                                   │
│ RegionServer #1  │◄──────────────────────────────────┼── 向 ZK 注册心跳
│ RegionServer #2  │                                   │
│ RegionServer #N  │                                   │
└──────────────────┘                                   │
└───────────────────────────────────────────────────────┘
```

**关键事实：**
- **读写不经过 HMaster**：Client 直接和 RegionServer 通信
- **HMaster 只负责元数据**：DDL（建表/删表）、Region 分配、负载均衡
- **Standby 在睡觉**：不干活，但醒着监听 ZK，随时准备抢锁

## 三、故障场景一：Active HMaster 进程崩溃

### 时序图

```
时间 ────────────────────────────────────────────────────────────────►

  Active HM          ZK 集群           Standby HM          结果
     │                  │                  │                    │
     │  会话保持        │  Ephemeral Node  │  Watcher 监听      │
     │◄──────────────► │  /hbase/master ──┼─────────────────►  │
     │                  │  [存在]          │  [等待中...]       │
     │                  │                  │                    │
  ═══╪══════════════════╪══════════════════╪════════════════════╪═══
     │                  │                  │                    │
     ▼  进程崩溃!       │                  │                    │
     │                  │                  │                    │
  [进程终止]            │                  │                    │
     │  无心跳发送      │                  │                    │
     │                  │                  │                    │
  ═══╪══════════════════╪══════════════════╪════════════════════╪═══
     │                  │                  │                    │
     │                  │  会话超时检测    │                    │
     │                  │  (默认 40 秒)    │                    │
     │                  │                  │                    │
  ═══╪══════════════════╪══════════════════╪════════════════════╪═══
     │                  │                  │                    │
     │                  │  删除临时节点    │                    │
     │                  │  /hbase/master   │                    │
     │                  │  [已删除] ──────►│  触发 Watcher!     │
     │                  │                  │                    │
  ═══╪══════════════════╪══════════════════╪════════════════════╪═══
     │                  │                  │                    │
     │                  │                  │  尝试创建          │
     │                  │◄─────────────────│  /hbase/master     │
     │                  │                  │  (竞争锁)          │
     │                  │                  │                    │
  ═══╪══════════════════╪══════════════════╪════════════════════╪═══
     │                  │                  │                    │
     │                  │  创建成功!       │                    │
     │                  │◄─────────────────│  我抢到锁了!       │
     │                  │                  │                    │
     │                  │  新会话建立      │                    │
     │                  │  Ephemeral Node  │                    │
     │                  │  [新 Active]     │                    │
     │                  │                  │                    │
  ═══╪══════════════════╪══════════════════╪════════════════════╪═══
     │                  │                  │                    │
  [已死]               │                  │  接管元数据管理     │
     │                  │                  │  • 加载表结构      │
     │                  │                  │  • 恢复 Region 分配│
     │                  │                  │                    │
     │                  │                  │  ┌──────────────┐  │
     │                  │                  │  │ 新 Active HM │  │
     │                  │                  │  │ (原 Standby) │  │
     │                  │                  │  └──────────────┘  │
     ▼                  ▼                  ▼                    ▼
```

### 关键时间点

| 阶段 | 耗时 | 说明 |
|------|------|------|
| 进程崩溃到无心跳 | 0s | 进程突然终止，不再发送心跳 |
| ZK 会话超时检测 | ~40s | 默认 `zookeeper.session.timeout` |
| 临时节点删除 | 即时 | 超时后 ZK 自动删除 Ephemeral Node |
| Watcher 触发 | < 1s | ZK 通知所有注册的 Watcher |
| Standby 竞争锁 | < 1s | 多个 Standby 同时尝试，只有一个成功 |
| 新 Active 初始化 | 数秒 | 加载元数据、恢复 Region 分配状态 |

### 影响范围

- **DDL 操作中断**：建表/删表等操作在切换期间失败
- **Region 故障恢复暂停**：RegionServer 挂了无法自动迁移（但已运行的 Region 不受影响）
- **数据读写不受影响**：Client 直接访问 RegionServer，不经过 HMaster
- **新节点无法加入**：RegionServer 启动时需要 HMaster 分配 Region

## 四、故障场景二：Active HMaster 网络分区（ZK 失联）

这个场景比进程崩溃更复杂，因为 **HMaster 进程还在运行**。

```
┌─────────────────────────────────────────────────────────────────────┐
│                          网络分区发生                                │
│                                                                     │
│   ┌───────────────────┐         ╳         ┌───────────────────┐    │
│   │   Active HMaster  │◄────断开────►│    ZK 集群        │    │
│   │   (进程还在运行!)  │         ╳         │                  │    │
│   └───────────────────┘               └───────────────────┘    │
│           │                                    │                   │
│           │                                    │                   │
│           │    我以为是网络抖动...               │  40秒后：没心跳    │
│           │    继续处理请求？                    │  删除临时节点!     │
│           │                                    │                   │
│           │                                    ▼                   │
│           │                            ┌───────────────────┐      │
│           │                            │ Standby HMaster   │      │
│           │                            │ Watcher 触发!     │      │
│           │                            │ 创建 /master 成功 │      │
│           │                            │ 成为新 Active!    │      │
│           │                            └───────────────────┘      │
│           │                                    │                   │
│           │    ┌───────────────────┐          │                   │
│           └───►│ 脑裂风险!         │◄─────────┘                   │
│                │ 两个 HMaster 同时 │                                │
│                │ 认为自己是 Active │                                │
│                └───────────────────┘                                │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### 为什么不会真的脑裂？

| 防护机制 | 原理 |
|----------|------|
| **ZK 独占锁** | 只有一个进程能在 ZK 上创建 `/hbase/master` 临时节点 |
| **旧 Active 的 Fencing** | 旧 HMaster 尝试操作 ZK 时发现会话失效，被迫退出 |
| **进程自毁** | 旧 HMaster 发现自己不再是 Active，主动关闭或降级 |

```
旧 Active HMaster 的自检流程：

1. 网络恢复后，尝试向 ZK 写入状态
2. ZK 返回 "会话已过期"
3. HMaster 检测到：我的锁没了！
4. 两种行为：
   a) 优雅退出（理想情况）
   b) 被 Fencing 机制强制杀掉（SSH 杀进程）
5. 系统中始终只有一个 Active HMaster
```

## 五、故障场景三：所有 HMaster 同时故障

```
┌─────────────────────────────────────────────────────────────────┐
│                    极端场景：所有 HMaster 全挂                    │
│                                                                 │
│   ZK 集群正常 ──► 但 /hbase/master 不存在                        │
│                    /hbase/backup-masters 为空                     │
│                                                                 │
│   ┌──────────────┐                                              │
│   │   Client     │── 数据读写? ──► 可以!                        │
│   └──────────────┘     (直接访问 RegionServer)                   │
│          │                                                      │
│          ▼                                                      │
│   ┌──────────────┐                                              │
│   │ RegionServer │── 继续服务已分配的 Region                      │
│   │   #1, #2, #N │── 向 ZK 发送心跳 (还活着)                     │
│   └──────────────┘                                              │
│                                                                 │
│   影响：                                                         │
│   ❌ 无法执行 DDL (建表/删表/修改表)                              │
│   ❌ RegionServer 故障时无法自动迁移 Region                        │
│   ❌ 无法执行负载均衡                                              │
│   ❌ 新 RegionServer 无法加入集群                                  │
│   ✅ 已有数据读写不受影响                                          │
│   ✅ Region 继续服务                                               │
│                                                                 │
│   结论：HMaster 不是单点故障，集群不会完全瘫痪                      │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## 六、关键配置参数

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `hbase.master.zookeeper.useACL` | false | 是否在 ZK 上使用 ACL 保护节点 |
| `zookeeper.session.timeout` | 40s | ZK 会话超时时间，决定故障检测速度 |
| `hbase.master.failover.wait` | 1s | 故障转移等待时间 |
| `hbase.zookeeper.quorum` | localhost | ZK 集群地址 |
| `hbase.zookeeper.property.clientPort` | 2181 | ZK 客户端端口 |

## 七、面试常问问题

### Q1: HMaster 挂了，HBase 还能读写数据吗？

> **能。** Client 直接和 RegionServer 通信，不经过 HMaster。HMaster 只负责元数据管理和 DDL 操作。但 RegionServer 故障时无法自动恢复。

### Q2: 为什么 HMaster 需要 Standby？

> 如果没有 Standby，Active HMaster 挂了之后：
> - 没人执行 DDL 操作
> - RegionServer 故障时 Region 无法迁移
> - 新节点无法加入集群
> - 负载均衡无法执行
>
> Standby 通过 ZK 的 Watcher 机制在秒级内完成故障转移。

### Q3: 什么是 Fencing？为什么需要它？

> Fencing（栅栏）是一种防止脑裂的机制。在网络分区场景下，旧的 Active HMaster 可能还在运行。Fencing 确保旧的 HMaster 无法继续操作，常见方式：
> - ZK 会话过期后旧 HMaster 操作 ZK 失败
> - SSH 远程杀掉旧进程
> - 依赖底层资源（如共享存储）的独占锁

### Q4: ZK 的 Ephemeral Node 在这起到了什么作用？

> Ephemeral Node 与创建它的会话绑定。当 HMaster 崩溃或网络断开时：
> - 会话过期 → 节点自动删除
> - 触发 Watcher → 通知 Standby 竞争
> - 天然实现了"谁活着谁当老大"的选举逻辑

## 八、与 RegionServer 故障处理的对比

| 维度 | HMaster 故障 | RegionServer 故障 |
|------|-------------|-------------------|
| **检测机制** | ZK Ephemeral Node | ZK Ephemeral Node |
| **检测对象** | `/hbase/master` | `/hbase/rs/<hostname>` |
| **处理者** | Standby HMaster | Active HMaster |
| **恢复动作** | Standby 成为 Active | Region 迁移 + WAL 重放 |
| **数据影响** | 无（不处理数据） | 无（WAL 保证不丢） |
| **服务影响** | DDL 中断 | 短暂读写中断（秒级） |
